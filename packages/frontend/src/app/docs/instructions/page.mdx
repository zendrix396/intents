          <h3>2.1 Latest Versions and Dependencies (as of late 2025)</h3>

          <h3>2.2 Code Structure within an Anchor Program</h3>

          <ul>
            <li>
              <strong>Inspiration:</strong>{" "}
              <code>
                assetCLI/programs/bonding-curve/programs/bonding-curve/src/
              </code>
            </li>
            <li>
              <strong>Best Practice:</strong>
              <ul>
                <li>
                  <code>lib.rs</code>: The main entry point. It should contain
                  the <code>#[program]</code> module and the public function
                  definitions for each instruction. Keep it clean and delegate
                  all logic to instruction-specific modules.
                </li>
                <li>
                  <code>instructions/</code>: A directory where each{" "}
                  <code>.rs</code> file implements the logic for a single
                  instruction. Each file contains the{" "}
                  <code>#[derive(Accounts)]</code> struct and the{" "}
                  <code>process</code> function.
                </li>
                <li>
                  <code>state/</code>: A directory where each <code>.rs</code>{" "}
                  file defines a program account (<code>#[account]</code>) or a
                  shared state struct.
                </li>
                <li>
                  <code>errors.rs</code>: A dedicated file for your{" "}
                  <code>#[error_code]</code> enum. This keeps all possible
                  failure states in one place.
                </li>
                <li>
                  <code>events.rs</code>: A file for defining on-chain events (
                  <code>#[event]</code>).
                </li>
                <li>
                  <code>constants.rs</code>: For any on-chain constants.
                </li>
              </ul>
            </li>
          </ul>

          <h3>2.3 State Management and PDAs</h3>

          <ul>
            <li>
              <strong>Program Derived Addresses (PDAs):</strong> All
              program-owned accounts should be PDAs. This allows your program to
              sign for them, creating "program-controlled" accounts.
            </li>
            <li>
              <strong>
                <code>init_if_needed</code>:
              </strong>{" "}
              For accounts that might need to be created on-demand (like an
              Associated Token Account or a user-specific data account),{" "}
              <code>init_if_needed</code> is the modern, preferred approach. It
              combines creation and access into a single, idempotent
              instruction.
            </li>
            <li>
              <strong>
                Account Sizing (<code>space</code>):
              </strong>{" "}
              Always define the space for your accounts. Use{" "}
              <code>8 + MyAccount::INIT_SPACE</code> where{" "}
              <code>INIT_SPACE</code> is derived using the{" "}
              <code>InitSpace</code> macro from Anchor on your struct. For
              variable-sized data like <code>String</code> or <code>Vec</code>,
              use <code>#[max_len(N)]</code> attributes as seen in{" "}
              <code>dmandate/programs/dmandate/src/state/user.rs</code>.
            </li>
            <li>
              <strong>
                Closing Accounts (<code>close = &lt;destination&gt;</code>):
              </strong>{" "}
              When an account is no longer needed, use the <code>close</code>{" "}
              constraint to reclaim the lamports locked for rent. This is a
              crucial practice for good on-chain hygiene.
            </li>
          </ul>

          <h3>2.4 Secure and Robust Instruction Logic</h3>

          <ul>
            <li>
              <strong>Validation First:</strong> The first thing every
              instruction handler should do is validate all inputs and account
              constraints. Use Anchor's <code>require!</code> macros with custom
              errors.
              <ul>
                <li>
                  <code>
                    require_keys_eq!(account1.key(), account2.key(),
                    MyError::InvalidAccount)
                  </code>
                </li>
                <li>
                  <code>
                    require!(amount &gt; 0, MyError::AmountMustBePositive)
                  </code>
                </li>
              </ul>
            </li>
            <li>
              <strong>Signer Checks:</strong> Anchor handles basic signer checks
              (<code>Signer</code> type), but always double-check that the
              correct party is signing, especially for privileged instructions.
            </li>
            <li>
              <strong>Custom Errors:</strong> Use a detailed{" "}
              <code>#[error_code]</code> enum.{" "}
              <code>
                assetCLI/programs/bonding-curve/programs/bonding-curve/src/errors.rs
              </code>{" "}
              is a great example. This makes debugging off-chain a thousand
              times easier, as you get a specific code instead of a generic
              transaction failure.
            </li>
            <li>
              <strong>Cross-Program Invocations (CPIs):</strong>
              <ul>
                <li>
                  Use Anchor's typed CPI interfaces (<code>CpiContext</code>).
                </li>
                <li>
                  Always use <code>invoke_signed</code> when calling from a PDA,
                  passing in the PDA seeds. The <code>get_signer_seeds()</code>{" "}
                  helper function in{" "}
                  <code>
                    assetCLI/programs/bonding-curve/programs/bonding-curve/src/instructions/curve/create_bonding_curve.rs
                  </code>{" "}
                  is a good pattern.
                </li>
              </ul>
            </li>
          </ul>

          <h2>Part 3: Off-Chain Development Best Practices</h2>

          <p>
            This is where the complex logic of your intent-centric layer will
            live.
          </p>

          <h3>3.1 The Rust Solver: Performance is Key</h3>

          <p>
            Your solver needs to be fast. It will likely perform many
            calculations, simulations, and data fetching. Rust is the ideal
            choice here.
          </p>

          <ul>
            <li>
              <strong>Asynchronous Everywhere with Tokio:</strong> All network
              calls (to the Solana RPC) and potentially CPU-intensive
              calculations should be asynchronous. Use <code>tokio</code> as
              your runtime.
              <ul>
                <li>
                  <strong>Libraries:</strong> <code>solana-client</code> (for
                  RPC communication), <code>solana-sdk</code>,{" "}
                  <code>tokio</code>, <code>serde</code>, and <code>clap</code>{" "}
                  (for the binary wrapper).
                </li>
              </ul>
            </li>
            <li>
              <strong>
                Error Handling with <code>anyhow</code> or{" "}
                <code>thiserror</code>:
              </strong>
              <ul>
                <li>
                  <code>thiserror</code>: For your solver library crate. It lets
                  you create detailed, custom error types.
                </li>
                <li>
                  <code>anyhow</code>: For your binary crate. It's great for
                  application-level error handling where you just need to bubble
                  up any error and display it.
                </li>
              </ul>
            </li>
            <li>
              <strong>Data Structures:</strong> Define clear, serializable
              structs for <code>Intent</code>, <code>ExecutionPath</code>, and{" "}
              <code>Solution</code>. Use <code>serde</code> to easily
              serialize/deserialize them.
            </li>
          </ul>

          <h3>3.2 TypeScript/Node.js for Services and Clients</h3>

          <p>
            For APIs, CLIs, and services that are less performance-critical,
            TypeScript offers faster development and a massive ecosystem.
          </p>

          <ul>
            <li>
              <strong>Latest Versions (as of late 2025):</strong>
              <ul>
                <li>
                  <code>@solana/web3.js</code>: Version <code>^1.98.x</code> or
                  newer.
                </li>
                <li>
                  <code>@coral-xyz/anchor</code>: Version <code>^0.31.x</code>{" "}
                  or newer.
                </li>
                <li>
                  <code>@solana/spl-token</code>: Version <code>^0.4.x</code> or
                  newer.
                </li>
              </ul>
            </li>
            <li>
              <strong>
                Project Structure (<code>assetCLI</code>):
              </strong>
              <ul>
                <li>
                  <code>src/services/</code>: Your wrapper classes (e.g.,{" "}
                  <code>bonding-curve-service.ts</code>).
                </li>
                <li>
                  <code>src/commands/</code>: If building a CLI, each file
                  defines a command.
                </li>
                <li>
                  <code>src/utils/</code>: Reusable helper functions (e.g.,
                  transaction sending logic, constants).
                </li>
                <li>
                  <code>src/types/</code>: Contains the auto-generated types
                  from your Anchor program.
                </li>
              </ul>
            </li>
            <li>
              <strong>Reliable Transaction Sending:</strong> Solana's network
              can be congested. Don't just{" "}
              <code>sendAndConfirmTransaction</code>.
              <ul>
                <li>
                  <strong>Best Practice:</strong> Build a robust{" "}
                  <code>sendTx</code> utility function as seen in{" "}
                  <code>assetCLI/src/utils/send_tx.ts</code>. This function
                  should:
                  <ol>
                    <li>
                      Use <code>getLatestBlockhash</code> right before sending.
                    </li>
                    <li>
                      Include <code>ComputeBudgetProgram</code> instructions to
                      request more compute units and set a priority fee. This is{" "}
                      <strong>essential</strong> for getting your transactions
                      included in a block.
                    </li>
                    <li>
                      Send the transaction with{" "}
                      <code>skipPreflight: false</code>.
                    </li>
                    <li>
                      Use a loop with a timeout to poll for the transaction
                      status using <code>getSignatureStatuses</code>.
                    </li>
                  </ol>
                </li>
              </ul>
            </li>
          </ul>

          <h2>Part 4: Tooling, Testing, and Deployment</h2>

          <p>Professional projects require professional tooling.</p>

          <h3>4.1 Comprehensive Testing Strategy</h3>

          <ul>
            <li>
              <strong>Rust Unit Tests:</strong> For pure logic in your Rust
              crates (e.g., a function that calculates the best swap route from
              a given set of data), write standard Rust unit tests (
              <code>#[cfg(test)]</code>).
            </li>
            <li>
              <strong>Anchor Integration Tests:</strong> Use Anchor's TypeScript
              testing framework. This is the primary way to test your on-chain
              programs. It spins up a local validator, deploys your program, and
              lets you execute transactions against it. The{" "}
              <code>tests/dmandate.ts</code> file is a perfect example.
              <ul>
                <li>
                  <strong>
                    Test Setup (<code>before</code> block):
                  </strong>{" "}
                  Airdrop SOL to test wallets, create necessary mints and token
                  accounts.
                </li>
                <li>
                  <strong>
                    Test Cases (<code>it</code> blocks):
                  </strong>{" "}
                  Test both the "happy path" (successful execution) and failure
                  cases (e.g., "fails to execute payment too early").
                </li>
              </ul>
            </li>
            <li>
              <strong>
                Local Development with <code>solana-test-validator</code>:
              </strong>
              <ul>
                <li>
                  <strong>Inspiration:</strong>{" "}
                  <code>assetCLI/scripts/local-dev.sh</code>.
                </li>
                <li>
                  <strong>Best Practice:</strong> Create a script that starts a
                  local validator and clones all the mainnet programs your
                  solver will interact with (e.g., Jupiter, Raydium pools,
                  Metaplex). This allows you to test your entire end-to-end flow
                  locally without spending real funds.
                </li>
              </ul>
            </li>
          </ul>

          <h3>4.2 CI/CD with GitHub Actions</h3>

          <ul>
            <li>
              <strong>Inspiration:</strong> The <code>.github/workflows/</code>{" "}
              files in the <code>assetCLI</code> and <code>dmandate</code>{" "}
              projects.
            </li>
            <li>
              <strong>Best Practice:</strong> Set up a CI pipeline that runs on
              every push. It should:
              <ol>
                <li>Install Rust, Node.js, and the Solana tool suite.</li>
                <li>
                  Run <code>cargo check</code> and <code>cargo clippy</code> on
                  your Rust code.
                </li>
                <li>
                  Build your Anchor program (<code>anchor build</code>).
                </li>
                <li>
                  Start a <code>solana-test-validator</code>.
                </li>
                <li>
                  Run your Anchor integration tests (
                  <code>anchor test --skip-local-validator</code>).
                </li>
              </ol>
            </li>
          </ul>
